Methodology

Instance Sourcing

Evaluation instances are mined from recent pull requests in Red Hat's open-source ecosystem using SWE-benchify, which produces Multi-SWE-bench-format instances from any Git repository with a test suite. Each instance consists of a bug-describing problem statement, a repository snapshot at the buggy commit, and a canonical test patch that exposes the bug.

The current instance set draws from Go repositories including Kubernetes, gRPC-Go, etcd, Podman, and others. Instances are selected based on the availability of a clean fail-to-pass test signal: at least one test must fail on the buggy commit and pass after the canonical fix.

Scaffold Design

Every model receives the same fixed system prompt instructing it to act as an expert Go software engineer. The prompt specifies a workflow (explore the repo, make targeted edits, verify compilation, call finish) and rules (do not modify test files, keep the fix minimal, do not add new external dependencies).

Models are evaluated through two scaffold paths depending on the model provider:

  • Vertex API scaffold (Anthropic models via Vertex Model Garden): The system prompt is passed as the API's system parameter with exactly two tools available: bash (run a shell command, 120-second timeout, 8K char output limit) and finish (submit the patch via git diff HEAD). The agent loop runs for up to 30 turns with a 4,096-token max per response.
  • Cursor CLI scaffold (GPT-series and some Anthropic model tiers): The Cursor agent CLI is invoked with --trust --yolo mode. The system prompt is concatenated into the user message, not passed as a system-level instruction. Cursor's agent uses its own built-in system prompt, its own richer toolset (file editing, code search, terminal, etc.), and has no explicit turn limit — only a 30-minute timeout.

The scaffold version, system prompt text, and tool definitions are pinned constants to ensure model comparisons are not confounded by harness variation. However, the two scaffold paths are fundamentally different environments (see Limitations).

Grading Pipeline

Grading is fully automated and Docker-isolated. For each attempt:

  1. Test-file rejection: If the candidate patch touches any *_test.go file or testdata/ directory, the attempt is marked as rejected_test_edit and scored as not resolved. This prevents reward-hacking by modifying the test assertions directly.
  2. Patch application: The candidate patch is applied to a fresh checkout at the base commit, then the canonical test patch is applied on top. Tests run against the canonical test surface, not any modifications the candidate might make to test infrastructure.
  3. Compilation check: The patched repository must compile with go build ./.... Compilation failure results in compiled = false and automatic failure.
  4. Test execution: The test suite runs with a 600-second timeout. Results are split into fail-to-pass (F2P) tests that must now pass and pass-to-pass (P2P) tests that must continue passing.
  5. Quarantine filtering: Known flaky or environment-dependent tests are quarantined and excluded from scoring. For etcd specifically, F2P is restricted to unit-test packages (integration and e2e tests are excluded).

An attempt is scored as resolved if and only if: the patch compiles, all F2P tests pass, and all P2P tests pass.

Metrics

  • pass@1: The fraction of individual attempts that fully resolve the issue. Each model runs 3 attempts (k=3) per instance with different seeds; pass@1 treats each attempt independently.
  • pass@3: The fraction of instances where at least one of the 3 attempts resolves the issue. This captures whether a model can solve a problem given multiple tries.
  • Cost per resolve: Total API/CLI spend divided by the number of resolved attempts (pass@1) or resolved instances (pass@3).
  • Compile rate: Fraction of Go attempts where the candidate patch compiles successfully.

Statistical Comparisons

Head-to-head model comparisons use McNemar's test (with continuity correction) on paired observations. In pass@1 mode, each attempt index on each shared instance forms a pair; in pass@3 mode, each shared instance is one observation. This paired design controls for instance difficulty.

Required sample sizes are computed using the Connor (1987) formula for McNemar's test, targeting 80% power to detect a 10-percentage-point difference. The dashboard reports whether the current instance count meets this threshold.

Limitations

Scaffold Confound

Key limitation: The two scaffold paths create a significant confound that makes cross-scaffold model comparisons unreliable.

When models run through the Cursor CLI scaffold, the eval's system prompt — including the critical instruction "Keep the fix minimal — change only what is necessary" — is demoted from a system-level instruction to part of the user message. Cursor's agent has its own system prompt and richer tool environment that may encourage more ambitious, multi-file refactors over minimal targeted fixes.

Empirically, we observe that Claude models through Cursor CLI tend to produce larger patches that change function signatures or make breaking API changes requiring updates to all callers. When they miss callers in large codebases (Kubernetes, gRPC-Go), the patch fails to compile. The same models through the Vertex API scaffold — where the "keep it minimal" instruction has system-level authority — produce smaller, backwards-compatible patches that compile more often.

Go-Only Compilation Checking

The compile-rate metric and compilation-as-prerequisite grading only applies to Go repositories, which have a well-defined go build ./... check. Instances in other languages (if added in the future) use a different grading pipeline that may not have an equivalent compilation gate.

Test-File Editing Prohibition

Candidate patches that modify test files are automatically rejected to prevent reward-hacking. While this is necessary for evaluation integrity, it may penalize legitimate fix approaches that require updating test helpers, fixtures, or test-adjacent code (e.g., shared test utilities that aren't in *_test.go files but live in testdata/).

Cost Measurement

Cost is measured differently across scaffolds. The Vertex API scaffold computes cost from token counts and published per-token pricing. The Cursor CLI scaffold captures cost from Cursor's usage event stream, which may include overhead not present in direct API calls. These differences mean cost comparisons across scaffolds are approximate.

Quarantine Filtering

Quarantined tests (known flaky or environment-specific) are excluded from grading. This filtering is necessary to avoid false negatives from infrastructure issues, but it means that a model could break a quarantined test without penalty. The set of quarantined tests is manually curated and may not be exhaustive.
